# Admin-Web 프리셋 품질 균일성 검증 설계

## 1. 목적

admin-web 프리셋이 서로 다른 복잡한 비즈니스 도메인에 대해 **균일한 품질**의 프로토타입을 생성하는지 검증한다.

기존 `resort-admin-spec.yaml`은 엔티티 3개, 요구사항 4개 수준으로 실제 고객 요구사항 대비 지나치게 단순하다. 실제 비즈니스 수준의 복잡도를 가진 eval spec 2개를 작성하고, 동일한 프리셋으로 순차 실행한 뒤 결과를 비교한다.

## 2. 검증 대상

- **프리셋:** `admin-web`
- **스펙 1:** E-commerce 주문/운영 관리 (`examples/ecommerce-admin-spec.yaml`, 팔레트: `corporate-blue`)
- **스펙 2:** HR/인사 관리 (`examples/hr-admin-spec.yaml`, 팔레트: `dark-modern`)

두 도메인은 의도적으로 다른 종류의 복잡도를 갖도록 설계했다:

| 복잡도 축 | E-commerce | HR |
|---|---|---|
| 상태 머신 수 | 3개 (주문/배송/반품) | 5개 (직원/연차/초과근무/평가/급여) |
| 참조 깊이 | 3단계 (주문→주문상세→상품옵션) | 3단계 (직원→부서→상위부서) |
| 자기참조 | 없음 | 부서 트리 |
| 승인 플로우 | 없음 | 연차/초과근무/평가 3개 |
| 역할 복잡도 | 단순 (보기/수정 제어) | 복잡 (부서 범위 제한, 본인 제한) |
| 자동 계산 | 쿠폰 할인, 소계 | 근무시간, 급여 산출 |
| 연동 복잡도 | 재고 차감/복원 | 연차 잔여일수 차감 |

E-commerce는 **트랜잭션 정합성**에서, HR은 **승인 워크플로우 + 범위 기반 권한**에서 프리셋을 다르게 압박한다.

### 3.2 역할 체계

| 역할 | 설명 | 제한 |
|---|---|---|
| admin | 전체 접근 | 없음 |
| manager | 주문/배송/반품 처리, 상품 조회 | 쿠폰 생성 불가, 고객 삭제 불가 |
| viewer | 전체 조회만 | 모든 생성/수정/삭제 불가, 버튼 미표시 |

### 3.3 상태 머신

**주문:** `주문접수 → 결제완료 → 상품준비 → 배송중 → 배송완료 → 구매확정` / `취소`는 배송중 이전에만 가능

- `주문접수 → 결제완료`: 결제 레코드 존재해야 전이 가능
- `결제완료 → 상품준비`: 모든 주문상세 항목의 재고 확인 후 차감
- `상품준비 → 배송중`: 송장번호 입력 필수
- 취소 시: 차감된 재고 복원, 결제 상태 → 환불처리

**배송:** `준비중 → 집화 → 이동중 → 배달중 → 배달완료` (역방향 전이 불가)

**반품:** `반품요청 → 수거중 → 수거완료 → 검수중 → 환불완료`

- `검수중 → 환불완료`: 환불액 입력 필수, 결제의 환불 상태 연동

### 4.2 역할 체계

| 역할 | 설명 | 제한 |
|---|---|---|
| admin | 전체 접근, 급여 확정 | 없음 |
| manager | 소속부서 직원 관리, 연차/초과근무 승인, 평가 작성 | 급여 확정 불가, 타 부서 직원 수정 불가 |
| employee | 본인 정보 조회, 연차/초과근무 신청, 자기평가 작성 | 타인 정보 조회 불가, 승인 불가 |

### 4.3 상태 머신

**직원:** `재직 → 휴직 → 재직` / `재직 → 퇴직` / `휴직 → 퇴직`

- `재직 → 퇴직`: 미처리 연차/초과근무 신청 있으면 경고
- `퇴직` 상태는 역전이 불가

**연차:** `신청 → 승인대기 → 승인 → 사용완료` / `반려`

- 잔여 연차일수 초과 시 신청 불가
- 승인자는 신청자의 부서장이어야 함

**초과근무:** `신청 → 승인대기 → 승인 → 정산완료` / `반려`

- 월 40시간 초과 시 신청 불가 (경고 표시)

**평가:** `미작성 → 자기평가완료 → 상사평가완료 → 조정중 → 확정`

- 자기평가 → 상사평가 순서 강제
- 확정 후 수정 불가

**급여:** `산출 → 검토중 → 확정 → 지급완료`

- 산출: 기본급 + 초과근무수당(승인된 것만) - 공제액 자동 계산
- 확정은 admin만 가능

## 5. 실행 계획

### 5.1 접근법

순차 실행 + 사후 비교 (접근법 A).

1. E-commerce 스펙 → `/fde-agent:run examples/ecommerce-admin-spec.yaml` → evaluator 결과 기록
2. HR 스펙 → `/fde-agent:run examples/hr-admin-spec.yaml` → evaluator 결과 기록
3. 두 결과물 수동 비교

순차 실행을 선택한 이유: 첫 번째 결과에서 프리셋 문제가 드러나면 두 번째 실행 전에 원인을 파악할 수 있고, 디버깅이 단순하다.

### 5.2 비교 기준

**evaluator 통과 (최소 기준):**

두 결과물 모두 NFR 4개(build, unit_test, page_check, console) 전부 pass.

**수동 비교 체크리스트 (균일성 기준):**

| 비교 항목 | 확인 내용 |
|---|---|
| 디자인 토큰 준수 | 두 결과물 모두 팔레트 색상만 사용하는가, 하드코딩 색상 없는가 |
| 컴포넌트 패턴 일관성 | 목록 페이지의 테이블/필터/페이지네이션 구조가 유사한가 |
| 디렉터리 구조 | rules/CLAUDE.md의 디렉터리 규칙을 동일하게 따르는가 |
| 자율 구현 품질 | 스캐폴드에 없는 패턴(트리 뷰, 승인 플로우, 일괄 처리 등)의 완성도 |
| 상태 머신 구현 | 전이 가드가 실제로 동작하는가, UI에 올바르게 반영되는가 |
| 역할 제어 | 역할별 버튼/메뉴 표시가 정확한가 |
| 코드 구조 | API 라우트, 타입 정의, 데이터 레이어 패턴이 유사한가 |
| 에러 처리 | 유효성 검사 실패, 관계 무결성 위반 시 사용자 안내가 적절한가 |

**한계 파악:**

어떤 요구사항에서 실패했는지, 그 원인이 프리셋의 구조적 한계인지 Builder의 역량 한계인지 분류한다.
